# Domain Models - ドメインモデル定義

**最終更新**: 2025年11月2日
**目的**: Revenue Intelligence Platform のドメイン知識とビジネスコンテキストの定義

---

## 📚 概要

このディレクトリには、プロジェクトのドメインモデル（ビジネスドメインの詳細定義）が格納されています。
**Phase 0（前提情報定義）** で作成され、Phase 1-3のダミーデータ刷新の基盤となった重要なドキュメントです。

### 2. [60商談_顧客プロフィール.md](60商談_顧客プロフィール.md)
**サイズ**: 27,254 bytes（844行）
**内容**: 60件の商談（SaaS 30社 + 不動産 30名）の詳細プロフィール

#### 各商談の定義項目
1. **基本情報**
   - 会社名/氏名
   - 業種/職業
   - 従業員数/年齢
   - 予算

2. **ビジネスコンテキスト**
   - 現在の課題（3-5項目）
   - 導入目的
   - キーパーソン（役職、名前）
   - 決裁プロセス

3. **競合状況**
   - 比較検討中の競合製品/サービス
   - 競合の強み・弱み
   - 自社の差別化ポイント

4. **アクションプラン**
   - 次のステップ
   - 提案内容
   - クロージング戦略

#### 活用シーン
- ダミーデータ生成（Deals, Stakeholders, Meetings）
- トランスクリプト生成時の商談コンテキスト
- AI分析機能のテストデータ

## 🎯 Phase 0での役割

### ダミーデータ刷新プロジェクト（Phase 0-3）

#### Phase 0: 前提情報定義（2時間）
このディレクトリのファイルを作成し、以降のPhaseの基盤を構築

#### Phase 1: 基盤データ刷新（3時間）
- Deals: ステージ再配分（業界標準準拠、成約率6.7%）
- Stakeholders: 全60件設定（100%カバレッジ）
- このディレクトリのプロフィールを基に生成

#### Phase 2: Transcript生成（4.5時間）
- 116ミーティングのtranscript生成
- このディレクトリの課題、予算、競合情報を活用

#### Phase 3: データ品質確認（30分）
- Phase 0→1→2の一貫性検証
- ドメインモデルとDB データの整合性確認

## 🔄 ドメイン駆動設計（DDD）との関係

### Ubiquitous Language（共通言語）
このディレクトリのプロフィールは、開発チーム・ビジネスチーム・AIが共有する**共通言語**として機能します。

### Bounded Context（境界づけられたコンテキスト）
- **SaaSドメイン**: 営業プロセス、商談管理、Revenue Intelligence
- **不動産ドメイン**: 物件紹介、内見、契約プロセス

### Domain Events（ドメインイベント）
- 商談ステージ変更
- ミーティング実施
- 競合比較
- 意思決定プロセス

## 💡 使用方法

### 新規商談データ追加時
1. このディレクトリのプロフィール形式に従って顧客情報を定義
2. 課題、予算、競合、キーパーソンを詳細化
3. `/database/seed.sql` または生成スクリプトで反映

### AI機能のテスト時
1. プロフィールから商談コンテキストを取得
2. Gemini APIリクエストに含めて分析精度を向上
3. 期待される分析結果と実際の結果を比較

### ドキュメント作成時
1. ペルソナ設定の参照
2. ユースケースシナリオの作成
3. デモ動画の台本作成
